# Topology CLI Flags — Reference & Justification

This document lists every command-line flag of `spike_benchmark` that
controls **network topology generation** (i.e. anything that changes the
sparse connectivity matrix that the storage formats are benchmarked on),
explains what it does, and justifies why it is required by the experimental
protocol described in the project proposal (Stetsenko, *Benchmarking Sparse
Matrix Storage Formats for SNN Spike Propagation*, Spring 2026).

The relevant code paths are
[src/main.cpp](src/main.cpp), [src/topology.cpp](src/topology.cpp), and
[include/topology.h](include/topology.h).

> Flags that only affect the LIF dynamics, output, or sweep orchestration
> (`--timesteps`, `--trials`, `--poisson-rate`, `--bg-current`,
> `--inject-rate`, `--gather-only`, `--output-csv`, `--subprocess`, …) are
> documented in [docs/README_running.md](docs/README_running.md) and are
> intentionally **not** repeated here — this file is restricted to the
> flags that determine *which graph* is generated.

---

## Single-run topology flags

### `--topology {er | fi | ba | ws}`

Selects the random-graph generator. Mapping to the proposal's "Topologies"
list (§4.2):

| Flag value | Generator | Proposal name |
|------------|-----------|---------------|
| `er` | `generate_erdos_renyi` | Erdős–Rényi |
| `fi` | `generate_fixed_indegree` | Fixed in-degree |
| `ba` | `generate_barabasi_albert` | Barabási–Albert (scale-free) |
| `ws` | `generate_watts_strogatz` | Watts–Strogatz (small-world) |

**Why it matters for the experiment.** Research Question 1 asks which format
wins *under varying topology*. The four generators cover the qualitatively
distinct degree-distribution regimes that the storage formats are sensitive
to:

* `er` — Poisson degree, statistical homogeneity (Brunel-style baseline).
* `fi` — deterministic in-degree → **zero ELLPACK padding waste**, the
  best-case scenario for ELL.
* `ba` — power-law degree → **worst-case ELLPACK padding** and load
  imbalance during scatter; stresses CSR/CSC asymmetry around hub neurons.
* `ws` — near-regular but with long-range rewires → tests cache locality
  when the column index pattern is *almost* banded.

Without all four, the format ranking cannot be claimed to be topology-robust
(which is the central claim of RQ1).

---

### `--size N`  *(int, default `1000`)*

Number of neurons, i.e. the matrix dimension `N × N`.

**Why it matters.** Drives matrix and working-set size. The proposal's
Step 2 grid is `N ∈ {1 000, 10 000, 50 000, 100 000}` and the central
hypothesis is that the format ranking changes once the working set spills
out of L1 → L2 → L3 → DRAM. `--size` is the knob that traverses that
hierarchy. It is also the independent variable for any throughput-vs-N
plot, including the size-scaling discussion in
[docs/README_cache_and_metrics.md](docs/README_cache_and_metrics.md).

---

### `--density p`  *(double, default `0.05`)*

Topology-dependent reinterpretation of a single "connection density" knob:

| Topology | Interpretation in `generate_topology()` | Resulting `nnz` |
|----------|------------------------------------------|-----------------|
| `er` | Edge probability *p* per ordered pair | ≈ N(N−1)·p |
| `fi` | Mean in-degree `K = ⌊p·N⌋` | exactly N·K |
| `ba` | Edges per new node `m = max(1, ⌊p·N⌋)` | m(m−1) + (N−m)·m |
| `ws` | Ring-lattice degree `K = max(2, ⌊p·N⌋)` | N·K |

**Why it matters.** The proposal explicitly sweeps `p ∈ {0.01, 0.05, 0.10,
0.20}` because (a) cortical connectivity sits in the 1–20 % sparse regime
(proposal §1) and (b) density determines whether a row of the matrix fits
in a single cache line, which in turn changes the predicted winner between
CSR and ELL. Reusing one flag across all four topologies (rather than
`--er-p`, `--fi-K`, …) keeps the sweep grid orthogonal — every (topology,
density) pair refers to the same target connection fraction, which is what
makes the cross-topology comparison fair.

---

### `--seed S`  *(unsigned, default `42`)*

Seed of the `std::mt19937` used by every generator (`er`, `fi`, `ba`, `ws`).

**Why it matters.** Step 6 of the proposal mandates "10 trials" with
`mean ± std`. Reproducibility requires that:

1. A given `(topology, N, density, seed)` always yields the **same** graph,
   so reported timings can be regenerated by reviewers.
2. Trial-to-trial variance reflects *runtime* noise, not graph noise — the
   benchmark harness keeps the seed fixed across the `--trials` loop and
   only re-randomises the spike drive.

Without a seed flag, neither requirement can be honoured.

## Sweep-mode topology flags

### `--sweep-sizes "N1 N2 …"`  *(default `"1000 5000 10000"`)*

Space-separated list of `N` values to iterate over. The current default
covers the lower three of the proposal's four target sizes; passing
`--sweep-sizes "1000 10000 50000 100000"` reproduces the full grid from
proposal §4.2.

**Why it matters.** This is the operational way to execute the
size-scaling axis of Step 6 without writing a shell loop, which keeps the
single CSV output schema-consistent across all rows.

---

### `--sweep-densities "p1 p2 …"`  *(default `"0.01 0.05 0.1"`)*

Space-separated list of densities. Defaults cover three of the four
proposal points; pass `"0.01 0.05 0.1 0.2"` for the full grid.

**Why it matters.** Sweeping density at fixed `N` is what isolates the
*memory-traffic-per-spike* axis from the *network-size* axis — the central
decomposition the format comparison relies on.

---

## Hard-coded topology parameters (and why they are not flags)

These are values that *do* affect the generated graph but are intentionally
fixed to keep the experimental design tractable.

| Parameter | Value | Where | Why it is not a flag |
|-----------|-------|-------|----------------------|
| Watts–Strogatz rewiring probability `β` | `0.3` | `generate_topology()` in [src/topology.cpp](src/topology.cpp) | The proposal calls for a single small-world point, not a β-sweep. β = 0.3 sits in the small-world plateau (high clustering, short path length), matching the canonical Watts & Strogatz (1998) regime. |
| ER edge weight | `1/√(N·p)` | `generate_erdos_renyi` | Brunel-style input normalisation; not part of the format-comparison axis. |
| FI edge weight | `1/K` | `generate_fixed_indegree` | Equal expected drive per neuron, isolating topology effects. |
| BA / WS edge weight | `1.0` | resp. generators | Weight magnitude does not affect sparse-format cost; only the sparsity pattern does. |
| Self-loop exclusion | always on | all generators | Standard convention; matches NEST/GeNN reference models. |

Exposing these as flags would expand the sweep grid without adding evidence
for any of the three research questions.

## How the flags map back to the research questions

| Research question (proposal §2) | Flags that operationalise it |
|---------------------------------|------------------------------|
| RQ1: Which format wins under varying size, density, topology? | `--format`, `--topology`, `--size`/`--sweep-sizes`, `--density`/`--sweep-densities` |
| RQ2: Do synthetic results hold on NEST-derived connectivity? | `--nest-csv` (overrides the topology generator with real connectivity, see [docs/README_csv_io.md](docs/README_csv_io.md)) |
| RQ3: Does the CPU ranking agree with GeNN GPU? | Same `--topology`, `--size`, `--density`, `--seed` values are mirrored by `scripts/genn_benchmark.py` so the two pipelines benchmark *the same graph instance*. |

`--seed` is the glue that makes RQ3 a like-for-like comparison — both the
C++ binary and the PyGeNN script must instantiate the same realisation
of an `(er, N=5000, p=0.05)` graph for their timings to be comparable.
